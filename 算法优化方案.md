# 照片识别与搜索算法优化方案

## 一、 现状问题诊断：为什么“然然玩黄色气球”搜不到气球？

根据对 `storage.py` 和 `search.py` 的分析，目前系统的搜索存在以下瓶颈：

1. **分词与 FTS 匹配的“噪音灾难”**：
   目前 `storage.py` 中的 `search_fts` 算法将用户的长句按照空格拆分后，使用 `jieba` 分词，并将所有分词结果用 `OR` 连接。
   例如：“然然玩黄色气球” 会被分词并转换为 `("然然"*) OR ("玩"*) OR ("黄色"*) OR ("气球"*)`。
   **后果**：数据库中所有包含“玩”、“黄色”的照片都会被召回。由于大量日常照片的 `scene` 或 `objects` 包含“黄色”或“玩”，导致真正包含“气球”的照片被海量的低质量 `OR` 匹配结果淹没。单独搜“气球”时，因为没有其他干扰词，所以能准确召回。
2. **LLM 的重排（Rerank）受限于极差的召回（Recall）**：
   目前 `search.py` 中的 LLM Rerank 是对召回的 Top 20 结果进行二次排序。由于第一步的 `OR` 检索没有把真正有“气球”的图片放在 Top 20 里，LLM Rerank 变成了“巧妇难为无米之炊”。
3. **LLM 的前处理仅仅是“同义词扩展”**：
   目前的 `--smart` 模式仅仅是让 LLM 生成同义词（如“气球”、“气球堆”），这实际上在 `OR` 逻辑下进一步加剧了“噪音灾难”。

## 二、 算法优化方案设计

为了让大模型真正起作用，并且提升本地算法的准确度，建议采用**“结构化查询 + 混合检索 (Hybrid Search)”**的策略。

### 方案 1：纯本地算法改造与 LLM 深度参与（不引入向量库，见效快）

改变 LLM “生成同义词” 的定位，让其成为 **“查询意图解析器 (Query Planner)”**。

**1. LLM 结构化意图解析**
不要让 LLM 续写同义词，而是让其将用户的自然语言翻译为结构化的检索要素。
例如输入：`"然然玩黄色气球"`
让大模型输出（JSON）：
```json
{
  "target_person": ["然然"], 
  "core_objects": ["气球"],
  "attributes": ["黄色"],
  "actions": ["玩"]
}
```

**2. 本地多级条件召回（抛弃纯 OR，使用 AND/OR 降级策略）**
根据 LLM 解析出的结构化信息，生成更精准的 SQLite 查询：
- **第一级（精确匹配）**：优先寻找 `persons` 表中匹配“然然”的 `image_id` 集合。
- **第二级（核心物体 AND 匹配）**：在这些图片中，强制要求 `scene` 或 `objects` **必须（AND）** 包含 `core_objects`（气球）。
- **第三级（属性加分）**：对于包含 `attributes`（黄色）和 `actions`（玩）的图片，在代码逻辑中给予分数加成，而不是用作宽泛的召回条件。

**优势**：确保搜索意图中的核心元素绝对不会丢失，同时保留大模型的意图理解能力。

### 方案 2：引入轻量级向量检索（彻底的语义搜索）

目前的纯文本 FTS 无法理解“狗”和“犬”的距离，也无法匹配复杂的空间关系。如果需要真正的“自然语言搜索”，必须引入 Embedding 向量。

**1. 向量化（Embedding）存储**
无需引入庞大的向量数据库（如 Milvus），针对本地应用，可采用以下极其轻量的方案：
- **方案 A（SQLite-Vec）**：直接为 SQLite 安装 `sqlite-vec` 扩展，使得 SQLite 本身支持存储和搜索向量。
- **方案 B（Numpy 纯内存计算）**：如果是个人照片库（万张级别），将每张图片的 `scene` + `objects` 用轻量的本地文本模型（如 `bge-small-zh`，几百MB）转换为向量，或者调用现有的大模型 API 获取 Embedding。将所有向量存为一个 `.npy` 或写入 SQLite Blob。查询时纯内存算 Cosine Similarity，几毫秒即可完成。

**2. 向量召回 + FTS 关键词召回（双路召回）**
- **路一（向量搜索）**：将用户查询“然然玩黄色气球”转化为向量，与数据库中所有图片的描述向量计算相似度，召回 Top 50。
- **路二（精确人物+FTS）**：提取人物“然然”，结合 FTS 搜索。
- **融合（RRF 倒数秩融合）**：将两路的得分进行 RRF 融合，得出最终的排序。

**优势**：彻底解决“形近词”、“语义关联”等问题，对长难句拥有极强的鲁棒性。

## 三、 推荐实施路径 (Todo)

综合考虑改造成本与收益，建议分两个阶段实施：

**阶段一：升级 LLM 意图解析与本地 AND/OR 逻辑（不加新库，解决当前痛点）**
1. 修改 `search.py`，将 `_llm_expand_query` 改为提取 JSON 结构化实体的 Prompt。
2. 升级 `storage.py` 中的 `search_fts`，接收结构化 JSON，分离 `人物`、`必须包含的物体` 和 `软性修饰词`。
3. 实现降级匹配：先尝试 `人物 AND 核心物体`，无结果再降级为宽泛匹配。

**阶段二：引入本地轻量级文本 Embedding**
1. 选用本地轻量 Embedding 模型（如 `sentence-transformers` 或 Ollama API）。
2. 在 `images` 表增加 `embedding` (BLOB) 字段。
3. 在录入/扫描图片时，顺便将其 `scene + objects` 计算向量并持久化。
4. 在搜索时，使用 numpy 余弦相似度计算匹配分，与 FTS 结果融合。
