# 人脸识别与管理功能设计方案

## 1. 总体目标
在现有的 `Photo_Identify` 图像信息语义扫描基础上，增加本地化、支持 GPU（5060Ti/CUDA）加速的多人脸识别与管理功能。类似于 iOS 照片的人脸相册，能自动聚类并允许用户打标签（重命名）。

## 2. 技术选型
*   **基础识别库**: 使用 **`insightface`** (基于 ONNX Runtime)。
    *   **理由**: 业界开源首选，精度极高（ArcFace），并且 `insightface` 底层通过 `onnxruntime-gpu` 可以完美调用 NVIDIA 5060Ti 显卡的 CUDA 核心进行加速，速度远超基于 dlib 的 `face_recognition`。
    *   **依赖项**: 
        ```toml
        # pyproject.toml / requirements.txt 新增
        insightface
        onnxruntime-gpu  # 必须，以利用 5060Ti
        scikit-learn     # 用于 DBSCAN 人脸特征聚类
        opencv-python
        ```
*   **聚类算法**: 使用 **DBSCAN** (Density-Based Spatial Clustering of Applications with Noise)。
    *   `Insightface` 提取出的人脸特征是 512 维的 float 向量 (embeddings)。
    *   使用 DBSCAN 计算余弦距离或欧氏距离，将相似度极高（如同一个人）的向量聚类成同一个 `person_id`。

## 3. 数据库结构修改 (`storage.py`)
需要使用 SQLite 存储提取出来的特征向量，并维护人脸与图片的映射关系。

新增表结构（利用现有 `db_path`）：
```sql
-- 1. 存储每一个检测到的独立人脸特征和在图中的位置
CREATE TABLE IF NOT EXISTS face_embeddings (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    image_id INTEGER NOT NULL,            -- 关联 images.id
    bbox TEXT NOT NULL,                   -- 人脸在图中的坐标 [x1, y1, x2, y2]
    embedding BLOB NOT NULL,              -- 512维特征向量的二进制序列化 (numpy.tobytes)
    cluster_id INTEGER DEFAULT -1,        -- 聚类分配的 ID (-1 为未分类或噪点)
    FOREIGN KEY(image_id) REFERENCES images(id) ON DELETE CASCADE
);

-- 2. 存储聚类后的人物实体（标签）
CREATE TABLE IF NOT EXISTS persons (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    cluster_id INTEGER UNIQUE,            -- 对应的 DBSCAN 分组号
    name TEXT NOT NULL,                   -- 用户自定义名称，默认可能是 "人物_01"
    cover_image_id INTEGER,               -- 封面图关联的 image_id
    cover_face_id INTEGER                 -- 封面图对应的 face_embeddings.id
);

-- 索引，加快通过图片找人脸和通过人脸找图片的查询速度
CREATE INDEX IF NOT EXISTS idx_face_image_id ON face_embeddings(image_id);
CREATE INDEX IF NOT EXISTS idx_face_cluster_id ON face_embeddings(cluster_id);
```

## 4. 核心逻辑修改

### 4.1 新增 `face_manager.py` (人脸核心处理层)
*   **硬件加速与回退检测**: 
    *   动态检测环境中是否安装了 `onnxruntime-gpu` 并且 `CUDAExecutionProvider` 可用。
    *   根据检测结果动态设置 `providers`，例如：如果 CUDA 可用，使用 `['CUDAExecutionProvider', 'CPUExecutionProvider']`；如果不可用，静默回退到 `['CPUExecutionProvider']`。
    *   暴露一个属性 `get_device_mode()` 返回当前的运行状态（如 "CUDA 加速" 或 "CPU 模式"），供 GUI 获取并在界面上展示。
*   **初始化**: 实例化 `insightface.app.FaceAnalysis`。
*   **提取特征**: `extract_faces(image_bytes)` -> 返回包含边界框和 embedding 的列表。
*   **聚类逻辑**: `cluster_all_faces()` -> 定期（或在扫描完毕后）从 `face_embeddings` 表读取所有未绑定的 embedding BLOB，转换为 Numpy 数组，运行 DBSCAN，然后更新他们的 `cluster_id` 并同步到 `persons` 表。

### 4.2 修改 `scanner.py` (扫描与写入分离控制)
*   入参新增 `enable_face_scan: bool = True`。
*   工作流分支逻辑：
    1.  如果这是一张**新图**：
        *   调用大模型进行语义信息扫描。
        *   如果 `enable_face_scan=True`，同时调用 `face_manager.extract_faces`。
        *   将新图信息写入 `images` 表。
        *   获取返回的 `image_id`，将提取到的人脸写入 `face_embeddings` 表。
    2.  如果这张图**已存在于 `images` 表中 (通过 MD5 命中)**：
        *   如果它已经有大模型的分析记录（跳过大模型调用）。
        *   如果它**没有**存在于 `face_embeddings` 表中，且 `enable_face_scan=True`，则**仅调用**人脸扫描并插入 `face_embeddings`。
        *   这也实现了“如果已经进行过信息提取，则直接进行人脸扫描”的需求。
*   在整个批量扫描结束后（所有 future completed 后），调用 `face_manager.cluster_all_faces()` 统一进行一次人脸聚类和分组运算，确保所有新增面孔分配到正确的人物组。

### 4.3 修改 `gui.py` (UI 层)
#### A. "信息扫描" 标签页
*   在开始扫描按钮上方，增加一个 `ttk.Checkbutton(text="同时进行人物扫描", variable=self.enable_face_scan_var)`。
*   在界面上增加一个 Label，通过调用 `face_manager.get_device_mode()`，动态显示当前的硬件加速状态（例如：`<span style="color: green">人脸引擎状态: GPU 加速 (CUDA)</span>` 或 `<span style="color: orange">人脸引擎状态: CPU 模式 (未检测到兼容显卡)</span>`）。
*   将该变量传入 `start_scan` 方法，并最终透传给 `scanner.py` 的 `scan()` 方法。

#### B. 新增 "人物管理" 标签页 (Tab 3, 顺延原 Tab 3 模型管理至 Tab 4)
*   **左侧/上部: 人物列表/网格**。
    *   读取 `persons` 表和对应封面图，展示（通过 `pillow` 从原图中根据 bounding box 裁剪出人脸作为头像缩略图）。
    *   双击/右键点击人物可重命名 (更新 `persons.name`)。
*   **右侧/下部: 属于该人物的照片详情**。
    *   选中左侧某个人物后，查询 `face_embeddings` -> `images`，将相关的所有照片以网格视图 (类似搜索页) 展现出来。
    *   支持切换上一张/下一张预览。

## 5. 执行顺序与风险提示
1.  **依赖安装环境**: 必须确保宿主机（用户环境）正确安装了 NVIDIA 显卡驱动以及兼容的 CUDA toolkit，否则 `onnxruntime-gpu` 可能会回退到 CPU 或报错。
2.  **聚类性能**: DBSCAN 在面对极大量数据（比如几十万人脸特征）时计算矩阵比较耗时。考虑到个人相册规模，使用 sklearn 默认实现足够，若数据非常庞大可考虑增量更新，但第一版采用全量内存聚类或简单的阈值匹配以保开发进度。
